# Conceptual Model

Hephaestus is grounded in **Cognitive Apprenticeship** (Collins, Brown & Newman, 1989) and built around a framework for practice-aware guidance. This page describes the core domain concepts, their relationships, the system architecture, and how both map to established learning theory.

## Domain Model

The domain model captures the core concepts Hephaestus works with. Seven entities form a pipeline: **Observe → Detect → Guide → Grow**.

<img alt="Hephaestus Domain Model" src="/img/diagrams/domain-model.svg" />

### The Seven Entities

| # | Entity | What it is | Key insight |
|---|--------|-----------|-------------|
| 1 | **Project** | Bounded endeavor with goals, timeline, and enrolled contributors | Sets the practice catalog that gives findings meaning |
| 2 | **Contributor** | Person (or AI agent) developing professional practice | The subject of observation and guidance |
| 3 | **Artifact** | Any evaluable output of a contributor's work | Pull requests, issues, reviews, comments, thesis chapters, design mockups |
| 4 | **Event** | Lightweight trigger recording that something happened | Same structure regardless of source platform |
| 5 | **Practice** | Named behavioral pattern with detection criteria | The vocabulary for talking about *how* people work |
| 6 | **Finding** | Immutable assessment result from evaluating an artifact against a practice | Verdict, severity, evidence, reasoning — separates assessment from coaching |
| 7 | **Guidance** | Adaptive coaching response grounded in findings | Closes the loop: Finding → Guidance → Contributor |

### Four Health Dimensions

We propose organizing practices into four dimensions so no single area dominates:

| Dimension | What it measures | Example practices |
|-----------|-----------------|-------------------|
| **Technical** | Contribution and review quality | Review thoroughness, anti-pattern detection rate, complexity management |
| **Process** | Workflow effectiveness | Contribution scoping, work-in-progress limits, issue linkage |
| **Social** | Collaboration quality | Review reciprocity, response time, cross-team engagement |
| **Cognitive** | Understanding and growth | Declining anti-pattern rate, reflection depth, self-regulation |

### Beyond Software Engineering (Future)

These examples illustrate how the entities *could* map to other domains — none have been validated yet:

| Entity | Software Engineering | Design | Research | Course Project |
|--------|---------------------|--------|----------|----------------|
| Project | Repository + Milestone | Design Brief | Study | Capstone |
| Contributor | Developer | Designer | Researcher | Student |
| Artifact | Pull request, review, issue | Mockup, critique, spec | Paper, experiment | Report, presentation |
| Event | "submitted pull request #42" | "uploaded mockup v3" | "submitted revision §4" | "gave peer feedback" |
| Practice | "Gives substantive reviews" | "Tests with users" | "Pre-registers hypotheses" | "Incorporates feedback" |
| Finding | Review lacks substance (severity: medium) | Mockup untested (severity: high) | Missing power analysis (severity: high) | Feedback not incorporated (severity: low) |
| Guidance | "Consider breaking this into smaller changes" | "How might color-blind users experience this?" | "Methods section lacks power analysis" | "What did you learn from the peer feedback?" |

## Architecture

<img alt="Hephaestus Architecture" src="/img/diagrams/architecture.svg" />

### Data Flows

1. **Observe**: Artifact Environment → Webhook Ingest → Message Broker → Application Server → PostgreSQL
2. **Detect (Practices)**: Application Server spawns Agent Sandbox → Agent Sandbox calls Application Server LLM proxy → LLM Provider → results persisted to PostgreSQL
3. **Guide (AI Mentor)**: Contributor → Webapp → Application Server → Intelligence Service → LLM Provider
4. **Interact**: Contributor → Webapp → Application Server (via User Management)

## Cognitive Apprenticeship Mapping

Hephaestus operationalizes the six methods of **Cognitive Apprenticeship** (Collins, Brown & Newman, 1989) plus **Fading** as a meta-principle:

| CA Method | Definition | Hephaestus Mechanism | Status |
|-----------|-----------|---------------------|--------|
| **Modeling** | Making expert practice visible | Positive-verdict findings surface good examples, leaderboard highlights what high performers do, achievement progression chains | Current |
| **Coaching** | Observing and providing feedback | AI mentor (Hattie feedback levels), practice detection findings with severity, inline comments on contributions | Current |
| **Scaffolding** | Providing support to perform beyond current ability | Draft contributions skip detection (reducing noise), cognitive load cap (max 5 negative findings per practice per job, max 100 per job) | Current |
| **Articulation** | Getting learners to externalize reasoning | Mentor reflection sessions, practice-defined contribution description quality checks | Current |
| **Reflection** | Enabling comparison with experts or peers | Contributor summaries, week-over-week trends, peer comparison via leaderboard, session history | Current |
| **Exploration** | Encouraging self-directed problem solving | Pull-based mentor (user initiates), hidden achievements | Current |
| **Fading** | Withdrawing support as competence grows | Contributor history per practice injected into agent context; agent selects CA method. No system-level fading logic yet. | Partial |

### Secondary Theories

| Theory | Role in Hephaestus |
|--------|-------------------|
| **Self-Regulated Learning** (Zimmerman, 2002) | Micro-theory for AI mentor design: forethought → performance → self-reflection cycle |
| **Model of Feedback** (Hattie & Timperley, 2007) | Feedback level hierarchy: Task > Process > Self-Regulation >> Self. Guides what the mentor and detector say |
| **Self-Determination Theory** (Deci & Ryan, 2000) | Engagement design: achievements → competence, team competitions → relatedness, pull-based mentor → autonomy |
