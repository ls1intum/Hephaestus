# Conceptual Model

This page describes the domain entities Hephaestus reasons about, the four channels through which feedback reaches people, the deployment architecture, and how each maps to the codebase. It is the canonical reference for anyone working on the platform.

## Domain Model

A **Workspace** defines a catalog of **Practices** and enrolls **Contributors**. Contributors produce **Artifacts** and emit **Events**. Practices are evaluated against artifacts (informed by events) to produce **Findings**. Findings are routed through **Channels** to the right audience. Contributors may **Contest** any finding — a first-class signal, not an afterthought.

### Entities

| Entity | What it is | Key insight |
|---|---|---|
| **Workspace** | The tenant. Owns a practice catalog, members, and feature toggles. | Practices live here, not on the platform — every workspace decides what "good" means. |
| **Contributor** | A person (or agent) participating in a workspace. | The subject of observation, the audience of guidance — never the object of ranking. |
| **Artifact** | A pull request, issue, review, or comment. | What practices are evaluated against. |
| **Event** | An immutable record of activity (commit pushed, comment posted, review submitted). | Same shape regardless of source platform; the input to detection. |
| **Practice** | A named, versioned, inspectable behavioural pattern with detection criteria. | The vocabulary for *how* people work — defined per-workspace, not vendor-shipped. |
| **Finding** | Immutable assessment of an artifact against a practice — verdict, severity, evidence, reasoning. | Findings are about the work, not the worker. |
| **Channel** | The route a finding takes to a person, with audience and feedback level. | The four channels are the design contribution; routing is structural, not cosmetic. |
| **Contestation** | A contributor's reaction to a finding (confirm, dispute, dismiss). | Contestation is *measured*, not just stored — disputes signal practice or detection drift. |

### The Four Channels

Channels are the load-bearing design choice in Hephaestus. Each finding is routed to the channel that fits the practice and the moment. Channels are aligned to feedback levels from the Hattie & Timperley taxonomy and direction is derived from self-determination theory (push limits autonomy; pull preserves it).

| Channel | Direction | Audience | Feedback level | Where it lives |
|---|---|---|---|---|
| **In-Context** | Push | Author of the artifact | Task | PR/MR comments and inline diff notes |
| **Reflection Dashboard** | Pull | Any contributor (private) | Process | Webapp profile/findings surface |
| **Conversational Mentor** | Dialogic | Contributor | Self-regulation | Webapp mentor surface |
| **Facilitator Dashboard** | Pull | Facilitator | Decision support | Webapp educator surface |

Routing is asymmetric on purpose. Author-side practices (e.g., *justifies changes with evidence*) push in-context. Reviewer-side practices (e.g., *leaves actionable change requests*) appear only on private reflection surfaces. There is no public ranking, by design.

### Mapping to Codebase

| Entity | Implementation | Status |
|---|---|---|
| Workspace | `Workspace` (with `WorkspaceFeatures`) | Implemented |
| Contributor | `User` + `WorkspaceMembership` | Implemented |
| Artifact | `Issue`, `PullRequest`, `PullRequestReview`, `PullRequestReviewComment` | Implemented |
| Event | `ActivityEvent` (immutable ledger) | Implemented |
| Practice | `Practice` (workspace-scoped catalog with detection prompt and category) | Implemented |
| Finding | `PracticeFinding` (verdict, severity, evidence, reasoning, guidance) | Implemented |
| Channel | Routing logic in `DiffNotePoster`, mentor handlers, and webapp routes | **Planned as first-class entity** |
| Contestation | `FindingFeedback` (confirm / dispute / dismiss with optional explanation) | Implemented |

Earlier revisions of this model treated Guidance as a co-equal entity. It is now folded into Finding (as the `guidance` text and a planned `recommendedAction` field) and into Channel (which carries the delivery semantics).

## Architecture

The deployment view shows the components that actually run; the practice-aware loop below shows the conceptual path from event to person.

### Deployment view

<img alt="Hephaestus Architecture" src="/img/diagrams/architecture.svg" />

| Component | Role |
|---|---|
| **Webapp** | Hosts the Reflection Dashboard, Conversational Mentor, and Facilitator Dashboard channels. |
| **Application Server** | Spring Boot. Owns the practice catalog, finding store, and routing. |
| **Webhook Ingest** | Adapts GitHub/GitLab webhooks into the activity event stream. |
| **Agent Sandbox** | Runs the practice-detection agent (Claude Code, OpenCode) in an isolated Docker container. |
| **Intelligence Service** | Powers the Conversational Mentor channel. |
| **User Management** | Keycloak realm with GitHub as identity provider. |
| **Message Broker** | Decouples webhook ingestion from downstream processing. |
| **Git Provider** | GitHub (primary) or GitLab (webhook ingestion + practice detection). |
| **LLM Provider** | Workspace-configurable. The Agent Sandbox and Intelligence Service each connect independently. |

### Practice-aware loop

<img alt="Hephaestus Practice-Aware Loop" src="/img/diagrams/practice-loop.svg" />

The loop the deployment diagram doesn't show: how activity becomes findings, and how findings reach people through the four channels. Contestation flows back from contributor to routing — a closed loop that supplies the data for measuring detection accuracy and practice-definition quality.
