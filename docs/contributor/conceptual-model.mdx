# Conceptual Model

The vocabulary the platform uses, and how each piece maps to the code. The reference if you're working on Hephaestus.

## Domain Model

A project enrols contributors and defines a practice catalog. Contributors author artifacts and make contributions on them. A practice requires certain contributions and produces findings. A finding is about a contributor, cites the contributions that supply its evidence, and the contributor reacts to it.

<img alt="Hephaestus Domain Model" src="/img/diagrams/domain-model.svg" />

### Entities

| Entity | What it is | Responsibilities |
|---|---|---|
| **Project** | The endeavour. Holds the practice catalog and members. | Enrol contributors; define and retire practices. |
| **Contributor** | A person (or agent) in the project. | Make contributions; react to findings. |
| **Artifact** | A pull request, issue, review, or comment. | The site of contributions. |
| **Contribution** | An act of contributing — a commit, a comment, a review. | The raw signal. |
| **Practice** | A named pattern with criteria for detecting it. | Detect findings; explain itself. |
| **Finding** | A verdict on a contributor's practice, with evidence and guidance. | What gets delivered back. |

### Relationships

| From | To | Verb | Cardinality |
|---|---|---|---|
| Project | Contributor | enrols | `1..* — *` |
| Project | Artifact | owns | `1 — *` |
| Project | Practice | defines | `1 — *` |
| Contributor | Artifact | authors | `* — *` |
| Contributor | Contribution | makes | `1 — *` |
| Contribution | Artifact | on | `0..1 — *` |
| Practice | Contribution | requires | `* — *` |
| Practice | Finding | produces | `1 — *` |
| Finding | Contribution | cites | `1..* — *` |
| Finding | Contributor | about | `* — 1` |
| Contributor | Finding | reacts to | `* — *` |

A finding is *about* the contributor whose practice it assesses — not about the artifact. The artifact is the substrate on which contributions happen; findings reach an artifact transitively, through the contributions they cite.

### Mapping to Codebase

The diagram uses conceptual names; the schema uses Hephaestus-specific names. They map 1:1.

| Conceptual | Implementation |
|---|---|
| Project | `Workspace` (with `WorkspaceFeatures`) — a `Workspace` may contain one or more `Repository` rows |
| Contributor | `User` and `WorkspaceMembership` |
| Artifact | `Issue`, `PullRequest`, `PullRequestReview`, `PullRequestReviewComment` |
| Contribution | `ActivityEvent` |
| Practice | `Practice` (per workspace, with detection prompt and category) |
| Finding | `PracticeFinding` (verdict, severity, evidence, reasoning, guidance) |

A finding's `guidance` text is delivered to the contributor as a comment alongside the existing review. A contributor's reaction (confirm, dispute, dismiss) is recorded by `FindingFeedback`.

## Architecture

What runs, and how a finding gets to a person.

### What runs

<img alt="Hephaestus Architecture" src="/img/diagrams/architecture.svg" />

| Component | Role |
|---|---|
| **Webapp** | Where you read findings, talk to the mentor, and look at your profile. |
| **Application Server** | Spring Boot. Holds the practice list and the findings; decides where each finding goes. |
| **Webhook Ingest** | Listens for GitHub and GitLab webhooks and turns them into events. |
| **Agent Sandbox** | Where each review actually runs — Claude Code or OpenCode in an isolated Docker container. |
| **Intelligence Service** | Powers the mentor. |
| **User Management** | Keycloak. GitHub signs you in. |
| **Message Broker** | Carries events between ingestion and processing. |
| **Git Provider** | GitHub or GitLab. |
| **LLM Provider** | Whichever model you choose. The agent and the mentor each connect on their own. |
