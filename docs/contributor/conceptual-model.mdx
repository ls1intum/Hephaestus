# Conceptual Model

The vocabulary the codebase uses, the channels feedback travels through, and how each maps to the code. The reference if you're working on Hephaestus.

## Domain Model

A workspace owns a list of practices and enrols contributors. Contributors produce artifacts and emit events as they work. Practices are read against the artifacts to produce findings. Findings travel to people through channels. People react to findings — a contestation, recorded.

<img alt="Hephaestus Domain Model" src="/img/diagrams/domain-model.svg" />

### Entities

| Entity | What it is | Key insight |
|---|---|---|
| **Workspace** | The tenant. Holds the practice list, members, and feature toggles. | Each workspace decides what "good" means. |
| **Contributor** | A person (or agent) in a workspace. | Who Hephaestus is reading and writing to. |
| **Artifact** | A pull request, issue, review, or comment. | What practices are read against. |
| **Event** | A record of something that happened — a commit, a comment, a review. | The raw signal Hephaestus reads. |
| **Practice** | A named pattern with criteria for detecting it. | The list each workspace writes for itself. |
| **Finding** | A verdict on a practice for an artifact, with evidence and reasoning. | What gets written back to the contributor. |
| **Channel** | The route a finding takes to a person. | How feedback travels: a comment, a dashboard, a conversation, an overview. |
| **Contestation** | A reaction to a finding — confirm, dispute, dismiss. | How we know whether a finding landed. |

### Channels

Each finding goes where it can do something useful: beside the work, on a dashboard you visit, in a conversation you start, or on the overview your facilitator reads.

| Channel | How it reaches you | Who reads it | Where it lives |
|---|---|---|---|
| **In-context** | Pushed to the work | The author of the change | PR/MR comments and inline diff notes |
| **Reflection** | You open it | You alone | Your profile in the webapp |
| **Conversation** | You start it | You alone | The mentor in the webapp |
| **Overview** | A facilitator opens it | A facilitator | The facilitator dashboard |

Author-side practices (like *describing your change*) reach the author of the change. Reviewer-side practices (like *leaving an actionable review*) appear on the reviewer's own profile. Facilitators see the picture across a team.

### Mapping to Codebase

| Entity | Implementation | Status |
|---|---|---|
| Workspace | `Workspace` with `WorkspaceFeatures` | Done |
| Contributor | `User` and `WorkspaceMembership` | Done |
| Artifact | `Issue`, `PullRequest`, `PullRequestReview`, `PullRequestReviewComment` | Done |
| Event | `ActivityEvent` | Done |
| Practice | `Practice` (per workspace, with detection prompt and category) | Done |
| Finding | `PracticeFinding` (verdict, severity, evidence, reasoning, guidance) | Done |
| Channel | Routing in `DiffNotePoster`, mentor handlers, and webapp routes | Coming as a first-class entity |
| Contestation | `FindingFeedback` (confirm / dispute / dismiss, optionally explained) | Done |

Older drafts had a separate Guidance entity. We've folded it into Finding (the text and a planned `recommendedAction`) and into Channel (the delivery).

## Architecture

What runs, and how a finding gets to a person.

### What runs

<img alt="Hephaestus Architecture" src="/img/diagrams/architecture.svg" />

| Component | Role |
|---|---|
| **Webapp** | Where you read findings, talk to the mentor, and look at your profile. |
| **Application Server** | Spring Boot. Holds the practice list and the findings; decides where each finding goes. |
| **Webhook Ingest** | Listens for GitHub and GitLab webhooks and turns them into events. |
| **Agent Sandbox** | Where each review actually runs — Claude Code or OpenCode in an isolated Docker container. |
| **Intelligence Service** | Powers the mentor. |
| **User Management** | Keycloak. GitHub signs you in. |
| **Message Broker** | Carries events between ingestion and processing. |
| **Git Provider** | GitHub today, GitLab in progress. |
| **LLM Provider** | Whichever model you choose. The agent and the mentor each connect on their own. |

### How a finding gets to a person

<img alt="Hephaestus Practice-Aware Loop" src="/img/diagrams/practice-loop.svg" />

Activity comes in, the agent reads it against the practice list, findings come out, and they travel through the right channel to the right person. Reactions flow back, and that's how we tell whether a finding landed.
