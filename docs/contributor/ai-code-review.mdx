Hephaestus runs an AI-powered code review pipeline that evaluates work against configurable software engineering practices. The pipeline reviews three artifact kinds — pull/merge requests, issues, and settled Slack conversation threads — each through its own `JobTypeHandler` (`PullRequestReviewHandler`, `IssueReviewHandler`, `ConversationReviewHandler`). When a student opens a non-draft MR (or uses `/hephaestus review`), the system detects relevant practices, runs an LLM agent inside a sandboxed container, parses the structured output, and posts observations as MR comments and inline diff notes. Issue and conversation reviews follow the same detection → sandbox → parse → deliver pipeline but with no diff and no inline notes (see [Practices](#practices) and [Delivery pipeline](#delivery-pipeline)).

## Pipeline overview

```mermaid

    participant GL as Git Provider

    AS->>SB: Create Docker container
    SB->>LLM: Evaluate practices (via LLM proxy)
    LLM-->>SB: Structured JSON observations
    SB-->>AS: Extract output files + logs
    AS->>AS: Parse, validate, filter, persist
    AS->>AS: Compose delivery (MR note + diff notes)
    AS->>GL: Post summary comment + inline diff notes
```

## Key components

| Component            | Class                            | Responsibility                                                                                                                                                                                  |
| -------------------- | --------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Detection gate       | `PracticeReviewDetectionGate`     | Ordered cheap-to-expensive checks: workspace resolution, draft skip, practices-enabled + trigger-mode settings, agent config, practice matching, `runForAllUsers` bypass, assignee presence, role-checker health, assignee role. Shared by `evaluate` (PR) and `evaluateIssue` (issue) |
| PR review handler    | `PullRequestReviewHandler`        | Context assembly (diff, metadata, practices), diff-scope filtering, delivery orchestration for pull/merge requests                                                                            |
| Issue review handler | `IssueReviewHandler`              | Issue counterpart -- no diff, no repo mount, no inline notes; posts a single issue comment                                                                                                     |
| Conversation handler | `ConversationReviewHandler`       | Slack-conversation counterpart -- repo-less; persists observations and publishes an event that drives the mentor DM delivery loop, nothing posts back to Slack                                |
| Result parser        | `PracticeDetectionResultParser`   | Parses agent JSON, validates and normalizes slugs, coerces `(presence, assessment, severity)` to the system's coherence invariants. Never throws -- failures go to `discarded` list            |
| Delivery composer    | `DeliveryComposer`                | Inline-first rendering: inlinable observations (with file locations) become compact MR summary entries + full diff notes; non-inlinable observations get full detail in MR summary            |
| Diff validator       | `DiffHunkValidator`               | Validates diff note line positions against actual diff hunks. Snaps invalid positions to nearest valid line (`TreeSet.floor`/`ceiling`)                                                        |
| Feedback service     | `FeedbackDeliveryService`         | Posts MR summary comment and diff notes to the git provider. Suppresses delivery for closed, merged, draft, or opted-out PRs                                                                   |
| Bot command          | `BotCommandProcessor`             | Listens for `/hephaestus review` comments (via Spring `@TransactionalEventListener`) to retrigger reviews                                                                                      |
| Job executor         | `AgentJobExecutor`                | NATS pull consumer: claims jobs with `SKIP LOCKED`, dispatches to sandbox executor pool, persists results in micro-transactions                                                                |

## Agent architecture

A single backend powers practice detection: the **Pi practice agent**, wired via `PracticePiAdapter` on top of the shared `PiRuntimeFactory`. One Pi session reads the diff/context once, then is driven through the matched practices in focused turns — one turn per practice area (a coherent 2-4 practice group), each turn reading only that area's per-practice criteria. The agent persists one structured observation per `report_observation` tool call as soon as it is ready; a coverage gate re-prompts, at the end of the turn loop, any practice no turn reported, so every matched practice gets an observation. The run-level summary is composed downstream from the persisted observations, not by the agent.

The runner (`pi-runner.mjs`, copied into the workspace as `.run-pi.mjs`) drives the Pi SDK in-process: the turn loop and coverage gate run under a soft-timeout steering message and a hard-timeout abort, followed by a format retry if the persisted output is incomplete. A top-level watchdog hard-exits the process at `AGENT_BUDGET_MS + 30s` so the orchestrator always observes a terminal state.

### Workspace layout

Every agent container gets this file structure (files vary by artifact kind — see notes):

The layout is split by **location**, not lore (ADR 0020): `inputs/` is read-only (the path-guard whitelists exactly this subtree), `work/` is writable scratch that is never collected, and `out/` is the only directory collected back into SQL. The constants live in `SandboxLayout`; see `docs/contributor/agent/workspace-abi.mdx` for the full ABI.

```
/workspace/
  inputs/                              # read-only — the path-guard whitelists exactly this subtree
    manifest.json                      #   telescope: integration-agnostic index (path/connector/sha256)
    sources/scm/repo/                  #   the SCM connector's source — git checkout (PR reviews only; RO mount)
    context/                           #   workspace context (WorkspaceContextBuilder populates this)
      metadata.json                    #     PR/issue metadata + commits (PR, issue)
      comments.json                    #     review/issue comments (PR, issue)
      diff.patch                       #     diff with [L<n>] annotations (PR only)
      diff_stat.txt                    #     git diff --stat output (PR only)
      diff_summary.md                  #     per-file diff chunks with index table (PR only)
      contributor_history.json         #     prior observations for this author, optional (PR only)
      general_comments.json            #     conversation-tab review discussion, optional (PR only)
      review_threads.json              #     review-decision + thread state, optional (PR only)
      linked_work_items.json           #     resolved linked-issue rows, optional (PR only)
      issue_summary.md                 #     issue body + comment thread rendered for the agent (issue only)
      project_inventory.json           #     cross-repo workspace inventory (issue, conversation; also PR)
      conversation_thread.json         #     ordered human turns of one settled Slack thread (conversation only)
    practices/                         #   per-practice catalog (generated from the DB)
      index.json                       #     [{slug, name, area}]
      {slug}.md                        #     per-practice criteria
      all-criteria.md                  #     all criteria bundled (reduces tool calls)
  work/                                # scratch the agent + precompute write; NEVER collected
    precompute/practices/{slug}.ts     #   precompute scripts (from DB, if present)
    precompute-out/                    #   precompute output (summary.md + per-practice JSON)
    analysis/practices/                #   directory for intermediate observation markers
  out/                                 # the ONLY directory collected back into SQL
  task.json                            # TaskEnvelope around Task.PracticeReview — prompt, jobId, workspaceId
  .pi/                                 # Pi SDK agent dir ($PI_CODING_AGENT_DIR)
    AGENTS.md                          #   Pi orchestrator instructions
    settings.json                      #   Pi SDK configuration (provider, model, compaction)
    extensions/                        #   custom provider extensions (auto-discovered)
  .run-pi.mjs                          # runner entry point (copied from the pi-runner.mjs classpath resource)
```

### Output schema

Each `report_observation` tool call persists one observation; the runner assembles the accumulated calls into a JSON object with an `observations` array (`result.json` / `AgentJob.output.rawOutput`):

```json
{
  "observations": [
    {
      "practiceSlug": "hardcoded-secrets",
      "title": "API key exposed in source",
      "presence": "PRESENT",
      "assessment": "BAD",
      "severity": "CRITICAL",
      "confidence": 0.95,
      "evidence": {
        "locations": [{ "path": "Config.swift", "startLine": 9, "endLine": 9 }],
        "snippets": ["private let apiToken = \"ghp_abc123\""]
      },
      "reasoning": "Hardcoded credential on +line...",
      "guidance": "Delete the line and use environment variables...",
      "suggestedDiffNotes": [
        {
          "filePath": "Config.swift",
          "startLine": 9,
          "endLine": 9,
          "body": "Delete this credential..."
        }
      ]
    }
  ]
}
```

**Presence** (`presence`): `PRESENT` (the practice's signal is in the change), `ABSENT` (it is missing), `NOT_APPLICABLE` (the practice does not apply to this change). **Assessment** (`assessment`): `GOOD` (a strength) or `BAD` (a problem) — required for `PRESENT`/`ABSENT`, omitted (null) for `NOT_APPLICABLE`.

**Severities**: `CRITICAL`, `MAJOR`, `MINOR`, `INFO` -- defined per practice in the criteria files.

**`subjectLogin`** (optional): for reviewer-audience practices on a multi-reviewer PR, the reviewer login the model proposes the observation is about. The server validates this against the resolved reviewer set before it can drive attribution — the model only proposes, it never decides.

## Practices

> **Terminology.** A **practice** is one detectable item (its slug, e.g. `scope-one-reviewable-change`). A **practice area** is one of the groupings that contain practices (entity `PracticeArea`, table `practice_area`). The canonical one-word name for the grouping is **area** (the field is literally `Practice.area` / `practice_area_id`; see `practice-feedback-schema.md` §2 and `practice-catalogue.md` — *area*, never *goal* or *category*). Spell it out as "practice area" only where a bare "area" would be ambiguous from context; never substitute "goal" or "category" for the grouping. (Unrelated uses — achievement categories, mentor goal-setting, CSS grid areas — are exempt.)

Practices are stored in the database (`practice` table, `criteria` column). At runtime, `PracticeCatalogInjector` generates `inputs/practices/{slug}.md` files from the DB criteria and injects them into the agent workspace. Each practice defines:

- What to look for
- Severity classification rules
- False-positive exclusions

The seeded default catalog (`resources/practices/default-catalog.json`) ships 35 practices across 12 practice areas; each new workspace is seeded with a copy on creation. Practices are fully configurable per workspace and can be added or modified without code changes.

## Delivery pipeline

After the agent returns observations, the server runs the delivery pipeline in `PullRequestReviewHandler.deliver()` (the PR path; `IssueReviewHandler` and `ConversationReviewHandler` run the same parse → coerce → persist → compose → post skeleton without the diff-specific steps):

1. **Parse** -- `PracticeDetectionResultParser` validates all fields, normalizes slugs (`toLowerCase` + replace `_` with `-`), and collects `suggestedDiffNotes` from `BAD` observations. Malformed entries are captured in a `discarded` list (never throws).

2. **Secret pre-pass** -- `SecretDiffScanner` runs a deterministic scan over the raw diff and injects synthetic `hardcoded-secrets` `PRESENT`/`BAD` observations for any hit the model didn't already flag at the same diff position, so a credential leak is caught even if the model abstains.

3. **Filter by diff scope** -- `filterByDiffScope` removes observations whose evidence locations don't intersect the actual diff (metadata-level and reviewer-audience practices are exempt). Prevents hallucinated observations about unchanged code; an all-`NOT_APPLICABLE` result over a non-empty diff is treated as a stale-diff failure rather than delivered as a false all-clear.

4. **Coerce coherence** -- `PracticeDetectionResultParser.coerceCoherence` re-derives `(presence, assessment, severity)` per observation: a defect-detector practice's `GOOD` becomes `NOT_APPLICABLE`, severity is nulled unless `assessment == BAD`, and an advisory (non-`BLOCKING_ELIGIBLE_PRACTICES`) `BAD` is capped to `MINOR` so craft feedback never presents as a merge-blocker.

5. **Persist** -- `PracticeDetectionDeliveryService.deliver` resolves the target (`PULL_REQUEST`/`ISSUE`/`CONVERSATION_THREAD`) and subject, and inserts each observation via `ObservationRepository.insertIfAbsent` (idempotent on `occurrence_key`), publishing `PracticeDetectionCompletedEvent`.

6. **Reviewer-audience firewall** -- reviewer-audience observations (`ReviewerAudiencePractices`, e.g. `reviews-substantively-with-understanding`) are persisted in step 5 so the reviewer's own dashboard/mentor sees them, but are filtered out here before anything posts back to the PR author. A `ReviewSubmitted` job is entirely reviewer-audience, so nothing posts.

7. **Compose** -- `DeliveryComposer` partitions the remaining author-audience observations into:
   - **Inlinable** (have file locations, not in internal paths like `inputs/context/`, practice not in `NON_INLINABLE_PRACTICES`) -- compact list in MR summary, full detail in diff notes
   - **Non-inlinable** (PR-description / commit-discipline practices such as `describe-what-and-why` and `commits-are-atomic-and-cohesive`, or no file location) -- full detail in MR summary
   - When all observations are `GOOD` strengths, composes a short approval comment naming the top strengths

8. **Validate positions** -- `DiffHunkValidator` parses the unified diff to extract valid new-side line numbers per file. Invalid positions are snapped to the nearest valid line (`TreeSet.floor`/`ceiling`).

9. **Post** -- `FeedbackDeliveryService` checks suppression conditions (PR closed, merged, draft, or author opted out) and, if not suppressed, posts the MR summary comment (with an HTML marker `<!-- hephaestus:practice-review:{jobId} -->` for identification) and inline diff notes to the git provider's API. On re-runs, `DiffNotePoster` first deletes old diff notes bearing the `<!-- hephaestus-diff-note -->` marker to prevent accumulation.

## Bot command

Students can type `/hephaestus review` in an MR comment to retrigger a review. The flow:

1. `GitLabNoteMessageHandler` detects the command prefix and publishes a `BotCommandReceivedEvent`
2. `BotCommandProcessor` listens asynchronously, validates the PR state, evaluates the detection gate, and submits a new review job

This uses Spring's event system to avoid a module dependency cycle between `integration.scm` and `agent`.

### Dev trigger

For development, enable the REST endpoint to manually trigger reviews:

```yaml
hephaestus:
  dev:
    trigger-enabled: true
```

Then trigger with:

```bash
curl -X POST "http://localhost:${SERVER_PORT}/api/dev/trigger-review?prId=123&workspaceId=1"
curl -X POST "http://localhost:${SERVER_PORT}/api/dev/trigger-review?issueId=123&workspaceId=1&triggerEvent=IssueClosed"
```

The port must match your `SERVER_PORT` environment variable (default: `8080` in `.env`).

## Adding a new practice

1. Insert a row in the `practice` table with all required fields: `slug`, `name`, `workspace_id`, and `trigger_events` (JSONB array of event names drawn from `TriggerEventCatalog`, e.g. `["PullRequestCreated", "PullRequestSynchronized"]` for a PR practice or `["IssueCreated", "IssueClosed"]` for an issue practice -- a practice may only subscribe to events matching its `applies_to`)
2. Set the `criteria` column with the evaluation criteria text (Markdown)
3. Link the practice to a **practice area** for grouping via `practice_area_id` (FK → `practice_area`, e.g. the seeded review-ready-work or testing-discipline areas)
4. No code changes needed -- `PracticeCatalogInjector` generates `inputs/practices/{slug}.md` from the DB criteria and the agent reads practices dynamically from `index.json`

### Wording the human-facing strings

`name`, `whyItMatters`, and `whatGoodLooksLike` are read by developers (and `whyItMatters` is posted verbatim as the "Why this matters" line on a real review comment), so keep them plain and human:

- **`name`** — a short imperative verb phrase (≤ 8 words), parallel with the others ("Scope the change to one concern").
- **`whyItMatters`** — 1–2 sentences: name the failure the habit prevents, then the payoff. No clichés, no hype.
- **`whatGoodLooksLike`** — one concrete sentence picturing the good end state, general across languages and stacks (no per-framework examples).
- Avoid AI-speak and clubby slang: drive em-dashes to near zero (prefer a period or comma); skip "ship", "hand off", "rubber-stamp", "leverage", "robust", "seamless"; gloss any unavoidable abbreviation on first use (WIP → work in progress). Keep precise domain terms (injection sink, idempotent, boundary) and let the sentence carry their meaning.
- `criteria` is machine-facing (the developer never sees it) — write it for detection accuracy, not for prose.

## Extending to new languages

The Pi orchestrator instructions (`pi-orchestrator.md`, mounted at `.pi/AGENTS.md`) are language-agnostic. Language-specific guidance lives in the practice criteria. To support a new language:

1. Write new practice criteria targeting the language's patterns and insert them in the `practice` table
2. The agent orchestrator file is language-agnostic -- no changes needed unless the language requires special analysis strategies
3. Practice criteria (in DB) may benefit from language-specific examples; precompute scripts (also in DB) often need per-language regex tweaks
