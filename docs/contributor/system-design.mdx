## Top-level architecture

The canonical architecture diagram lives in [`docs/diagrams/architecture.hyl`](https://github.com/ls1intum/Hephaestus/blob/main/docs/diagrams/architecture.hyl) — see the [Conceptual Model](/contributor/conceptual-model#architecture) for the rendered version and data flow descriptions.

The legacy StarUML diagram below predates the canonical version and covers the same ground at a lower level of abstraction:

![Hephaestus top level architecture](./img/system-design/top_level_architecture.svg)

The platform is intentionally layered. Internally we distinguish between:

- **Application Server** — Spring Boot, integrates with code platforms, manages the leaderboard, and exposes the primary REST API.
- **Intelligence Service** — TypeScript + Hono guidance logic that powers the AI mentor and produces conversational artifacts.
- **Webhook Ingest** — Event ingestion for GitHub/GitLab webhooks, decoupled from downstream processing via a message broker.
- **Agent Sandbox** — Isolated Docker containers running AI agents (Claude Code, OpenCode) for practice detection.
- **User Management** — Keycloak identity realm configured with GitHub as identity provider.

Externally we integrate with code hosting (GitHub and GitLab), Slack notifications, and the analytics stack used for anonymised usage metrics. Data enters through webhooks, gets enriched via API calls, and surfaces in the React client.

## Engagement & recognition model

![Engagement analysis object model](./img/system-design/gamification_analysis_object_model.svg)

The analysis object model maps webhook payloads to Hephaestus entities. Core classes (`Review`, `PullRequest`, `Contributor`) mirror code platform data, while `Leaderboard`, `Reward`, and `Dashboard` extend the model for competitions.

The original webhook events contain redundant fields, so we normalise them into dedicated entities. Around that core we layer abstractions for leaderboards, rewards, and dashboards that keep engagement state cohesive and queryable.

## AI mentor architecture

![LangGraph structure for AI mentor](./img/system-design/langgraph.png)

The mentor relies on LangGraph nodes that update storage, generate responses, and maintain state. Conditional edges decide which node executes next based on the conversation context.

Each call to the mentor executes the graph: storage update nodes persist longitudinal context, response nodes craft user-facing replies, state nodes mutate the TypedDict that represents the conversation, and conditional edges steer the flow. The result is a predictable yet adaptive guidance experience.

## Core workflows

### Login and authorisation

![GitHub login and authorisation flow](./img/system-design/login_authorization_flow.svg)

The login flow leverages GitHub OAuth via Keycloak. Successful login grants access tokens for both the client and application server.

### New review activity

![New review activity workflow](./img/system-design/new_review_activity.svg)

Webhook events trigger data ingestion via the webhook ingest service. The application server enriches the data and recalculates leaderboard standings in real time.

These workflows highlight the two critical loops: authenticating via GitHub (producing application + mentor tokens) and streaming review events through ingest → enrichment → leaderboard updates. Keeping these diagrams current avoids surprises when we extend integrations.
