## Top-level architecture

![Hephaestus top level architecture](./img/system-design/top_level_architecture.svg)

The platform is intentionally layered. Internally we distinguish between:

- **Application Server** - Spring Boot, integrates with Git providers, records activity events and practice observations, exposes the primary REST API, and runs the Pi mentor agent in-process (streaming to clients via SSE).
- **Webhook Server** - same image as Application Server, activated with `SPRING_PROFILES_ACTIVE=prod,webhook` so only the inbound webhook receiver beans load. Restart-independent from Application Server; publishes to the same NATS JetStream streams that the sync consumer reads. See ADR 0008.
- **User Management** - Hephaestus-native auth: Spring Security `oauth2Login` federates to GitHub / gitlab.lrz.de, then issues a cookie-session JWT (ADR 0017). Accounts + federated identity links live in the application database.

Externally we integrate with code hosting (GitHub today, GitLab planned), Slack/email notifications, and the analytics stack used for anonymised usage metrics. Data enters through webhooks, gets enriched via API calls, and surfaces in the React client.

## Analysis object model

The analysis object model maps GitHub webhook payloads to Hephaestus entities. Core classes (`Review`, `PullRequest`, `Developer`) mirror GitHub data, and around that core we layer the append-only `ActivityEvent` log plus the practice observation entities that drive achievements and practice reflection.

The original webhook events contain redundant fields, so we normalise them into dedicated entities that keep activity and practice state cohesive and queryable.

## AI mentor architecture

![LangGraph structure for AI mentor](./img/system-design/langgraph.png)

The mentor relies on LangGraph nodes that update storage, generate responses, and maintain state. Conditional edges decide which node executes next based on the conversation context.

Each call to the mentor executes the graph: storage update nodes persist longitudinal context, response nodes craft user-facing replies, state nodes mutate the TypedDict that represents the conversation, and conditional edges steer the flow. The result is a predictable yet adaptive mentoring experience.

## Core workflows

### Login and authorisation

![GitHub login and authorisation flow](./img/system-design/login_authorization_flow.svg)

The login flow uses GitHub (and optionally gitlab.lrz.de) OAuth directly. On success the server mints a short-lived ES256 cookie-session JWT; the SPA reads `GET /user` (no token in the browser).
