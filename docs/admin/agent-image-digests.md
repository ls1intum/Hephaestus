The `agent-pi` Docker image runs every practice-review sandbox and every Pi mentor container. Production references it by **sha256 digest** so the bytes can't change under a running deploy and GHCR retention can't break sandboxes between releases. This page explains where the pin lives, how the release workflow keeps it current, and how to roll back.

## Prerequisites

Before relying on this system, verify once per environment:

1. **Docker Compose v2.24 or newer.** `compose.app.yaml` uses the long-form `env_file: [{path, required}]` syntax shipped in Compose v2.24 (Jan 2024). Older hosts must upgrade.
2. **GHCR retains versioned tags.** Each release tags the agent-pi manifest as `:vX.Y.Z`. As long as those tags exist, the digest is retained. Confirm GHCR isn't configured to prune versioned tags on the `ls1intum/hephaestus/agent-pi` package.
3. **Coolify preserves the `env_file:` directive** in `docker/compose.app.yaml`. The pin only flows into the app server if Coolify keeps the directive on deploy. Validate by setting the file to a sentinel value and `docker compose exec application-server env` after a deploy.
4. **No one sets `HEPHAESTUS_AGENT_IMAGE_REFERENCE` in either the Coolify UI or `compose.app.yaml`'s `environment:` block.** Compose precedence is `environment:` > `env_file:`, and shell-interpolated `${VAR}` from the deploy host shadows both. Treat the variable as owned by the pin file alone.

## Source of truth: `docker/agent-image-pin.env`

```bash
HEPHAESTUS_AGENT_IMAGE_REFERENCE=ghcr.io/ls1intum/hephaestus/agent-pi@sha256:<digest>
```

`docker/compose.app.yaml` loads it for the `application-server` service via `env_file:` (with `required: false` so a missing file still bootable in dev). The application server reads the value into `hephaestus.agent.image.reference`; `application-prod.yml` sets `hephaestus.agent.image.require-digest: true`, and `AgentImagePinGuard` refuses to start on a tag reference.

**Do not edit this file by hand.** The release workflow rewrites it on every release.

## How the pin gets updated

`.github/workflows/release.yml` does this for every semver release:

1. **Retag**: `docker buildx imagetools create -t agent-pi:vX.Y.Z agent-pi:<sha>`.
2. **Verify**: compute the OCI manifest digest from `--raw | sha256sum` for both source and target tags; assert they match. Catches any registry-side rewrite.
3. **Smoke**: pull the digest, exercise the real entrypoint, spot-check `bun` and `node` binaries.
4. **Commit**: rewrite the `HEPHAESTUS_AGENT_IMAGE_REFERENCE=…` line and push a `chore(release): pin agent-pi to <digest> for vX.Y.Z [skip ci]` commit to `main` (race-tolerant: rebase-and-retry up to 3 times).

Images are cosign-signed (keyless OIDC) and provenance-attested in the build workflow, then verified in the same job to catch OIDC misconfiguration immediately.

## Verification

```bash
# Pull the current pinned digest out of the env file:
DIGEST=$(grep -oE '@sha256:[a-f0-9]{64}' docker/agent-image-pin.env)
IMAGE=ghcr.io/ls1intum/hephaestus/agent-pi${DIGEST}

cosign verify "$IMAGE" \
  --certificate-identity-regexp '^https://github.com/ls1intum/hephaestus/' \
  --certificate-oidc-issuer 'https://token.actions.githubusercontent.com'

gh attestation verify "oci://$IMAGE" --owner ls1intum
```

The git log is the human view of the pin history:

```bash
git log --oneline -- docker/agent-image-pin.env
gh release view vX.Y.Z
```

## Rollback procedure

If a pinned digest breaks in production:

```bash
git log --oneline -- docker/agent-image-pin.env
git revert <bad-release-commit>
git push origin main
```

Coolify auto-deploys from `main`; new sandboxes pull the rolled-back digest. In-flight mentor containers keep their already-pulled image until they restart. Under two minutes end-to-end.

**Note**: the broken `:vX.Y.Z` tag on the registry still points to the bad digest — `git revert` only moves what `main` references. If your deploy pipeline pins by version tag in another place, or if you can't accept that the bad tag still exists, cut a `vX.Y.Z+1` patch release once the fix is committed.
